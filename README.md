# Smart Meter Energy Analytics

**Profiling electricity demand across 1.26M hourly smart-meter readings: when the grid peaks, how weather drives load, behavioural customer segments, faulty-meter detection, and a short-term demand forecast.**

🔗 **Live dashboard:** _add your GitHub Pages link here once published_

> **Data disclosure:** the dataset is **synthetic**, generated by [`analysis/generate_data.py`](analysis/generate_data.py) (144 meters × 8,760 hours = 1,261,440 readings). Every figure is the *real output* of the pipeline. The methodology transfers directly to real AMI (advanced metering infrastructure) data.

---

## The question

Given a year of hourly meter data for 144 customers, characterise the load: identify peak demand and when it happens, quantify how temperature drives consumption, segment customers by behaviour, catch malfunctioning meters, and forecast near-term demand for capacity planning.

## Headline findings

| Metric | Value |
|---|---|
| Meters / readings | 144 / 1,261,440 |
| Total demand (2024) | 4,066 MWh |
| System peak demand | 890 kW, at 09:00 |
| Load factor (avg ÷ peak) | 0.52 |
| Temperature–demand correlation | −0.78 (heating-dominated U-curve) |
| Faulty meters detected | 6 of 6 — recall 100%, precision 100% |
| Forecast accuracy (holdout) | 96.4% (MAPE 3.6%) |

- **The system peaks mid-morning** as commercial and industrial load ramps up, with a secondary residential evening rise.
- **Demand is heating-dominated**: consumption is lowest in the mild temperature band and climbs sharply in the cold, with a smaller cooling rise at the hot end — the classic U-shaped response.
- **KMeans on behavioural features** (load timing, load factor, weekend ratio) splits the fleet into **low / medium / high** usage tiers of 85 / 34 / 19 meters, which line up closely with residential, commercial, and industrial customers.
- **Robust (median/MAD) anomaly detection** flagged all six malfunctioning meters — those with zero-reading dropouts or implausible spikes — with no false positives.
- **A Holt-Winters forecast** of daily system demand tracks a held-out month to 96.4% accuracy and projects 14 days forward.

## Methods

- **Load profiling** — average demand by hour of day to locate the system peak.
- **Temperature-response analysis** — daily demand vs temperature, binned into the U-curve, plus correlation.
- **KMeans segmentation** — standardised behavioural features; faulty meters excluded from clustering.
- **Robust anomaly detection** — median/MAD-style thresholds on zero-share and spike ratio, scored against ground truth.
- **Holt-Winters** — additive triple exponential smoothing with weekly seasonality, **implemented from scratch** (no statsmodels dependency); parameters chosen by holdout MAPE.

## How it's built

```
analysis/
  generate_data.py    # 1.26M-reading synthetic AMI dataset
  analysis.py         # KPIs + load profile + U-curve + KMeans + anomaly + forecast -> JSON
  build_dashboard.py  # embeds the JSON into a self-contained dashboard
sql/
  schema.sql          # meter + readings warehouse schema
  analysis_queries.sql# KPIs, load profile, monthly, 7-day moving avg (window fn), anomaly candidates
data/
  meters.csv          # meter metadata
  readings_sample.csv # 14-day sample (the full readings.csv is regenerated by the script)
docs/index.html       # the dashboard (served by GitHub Pages)
```

**Stack:** Python (pandas, scikit-learn) · SQL (window functions) · Chart.js · GitHub Pages.

## Run it yourself

```bash
pip install pandas numpy scikit-learn
python analysis/generate_data.py     # writes data/*.csv (full readings.csv ~46MB)
python analysis/analysis.py          # writes docs/dashboard_data.json + prints findings
python analysis/build_dashboard.py   # writes docs/index.html
```

> The full `data/readings.csv` (1.26M rows) is generated locally and git-ignored to keep the repo light; a 14-day `readings_sample.csv` is committed so the data shape is visible.
